## 1. Что это

**Soroka** — Telegram-бот, который превращает «Избранное» в персональную базу знаний. Канал-инбокс собирает любой мусор, который пользователь форвардит/скидывает в течение дня; DM с ботом — поисковая строка по этому мусору, возвращающая **оригинальные ссылки и файлы**, а не пересказы.

**Ключевые принципы:**

1. **Никаких галлюцинаций.** Бот возвращает оригинальный источник, а не сгенерированный ответ.
2. **Минимум LLM-токенов.** Эмбеддинги и поиск — детерминированные. LLM работает только в двух местах: разбор намерения запроса и реранкинг топа. Возможность выбрать `:free` модель.
3. **Простая раздача аудитории.** Self-hosted на дешёвом EU VPS, один `docker compose up`, конфиг-через-Telegram.
4. **Готовность к MCP.** Архитектура изначально такая, что база данных доступна агентам (Claude Code, Cursor) через MCP-сервер по SSH-stdio.
5. **Surgical changes only.** Нет тэгов, нет таксономии, нет «топиков» — это упрощает и пользователю, и коду.

## 2. Пользовательский путь

### 2.1. Сохранение (канал «Избранное 2»)

Пользователь приватно владеет каналом «Избранное 2». Бот добавлен в канал админом и видит каждое сообщение. Любая публикация в канале → запись в базу.

**Что бот принимает:**

| Тип | Что извлекается |
|-----|-----------------|
| Текст | Сам текст |
| Голос (`voice`) | Транскрипт через Deepgram |
| YouTube-ссылка | Транскрипт через `yt-dlp --write-auto-sub` (если нет — заголовок+описание видео) |
| Веб-ссылка | Статья через `trafilatura` |
| PDF | Текст через `pypdf` |
| DOCX | Текст через `python-docx` |
| XLSX | Содержимое листов через `openpyxl` |
| Картинка | OCR через `tesseract` (`rus+eng`), плюс caption если есть |
| Файл >20MB | Только метаданные (имя, тип, caption), помечается ⚠️ |

**Реакции на сообщение в канале (Telegram bot reactions):**

- 🔄 — бот начал обработку
- ✅ — успешно сохранено в базу
- ❌ — ошибка (детали логируются)
- ⚠️ — файл >20MB, сохранены только метаданные

### 2.2. Поиск (DM с ботом)

Пользователь пишет боту в личку — текстом или голосом. Голос распознаётся Deepgram → дальше как текст.

**Алгоритм:**

1. Запрос → LLM (intent detection): что ищем, какие фильтры (тип контента, период)
2. Гибридный поиск:
   - BM25 через SQLite FTS5 → top-30
   - Dense через Jina v3 + `sqlite-vec` → top-30
   - RRF (Reciprocal Rank Fusion) объединяет → top-15
3. LLM-реранкер выбирает top-5 наиболее релевантных
4. Бот отдаёт пять блоков:
   ```
   📌 [тип] заголовок
   <ссылка на оригинал в Избранном 2>
   фрагмент по запросу 30-50 слов

   ─────
   ```

**Если ничего не найдено** — бот честно пишет «не нашёл» и предлагает уточнить запрос.

### 2.3. Команды бота

| Команда | Что делает |
|---------|------------|
| `/start` | Мастер настройки из 6 шагов (см. §3.8) |
| `/help` | Краткая справка по всему |
| `/status` | Текущий ключ, модели, количество записей в базе, использование квоты, статус GitHub-зеркала |
| `/setjina` | Заменить ключ Jina |
| `/setdeepgram` | Заменить ключ Deepgram |
| `/setkey` | Заменить ключ OpenRouter |
| `/models` | Inline-кнопки для выбора основной/fallback модели (с пагинацией) |
| `/setgithub` | Заменить GitHub-токен и/или имя репо-зеркала |
| `/setvps` | Заменить IP/юзера VPS (используется в команде `/mcp`) |
| `/setinbox` | Сменить канал-инбокс (нужен forward из нового канала) |
| `/export` | Экспорт всей базы (см. §3.5) |
| `/mcp` | Распечатать готовый JSON-конфиг для Claude Code/Cursor |
| `/skip` | В мастере `/start` — пропустить опциональный шаг (только GitHub-зеркало) |
| `/cancel` | Отменить текущий мастер/диалог настройки |

### 3.2. Схема базы данных (SQLite)

```sql
-- Владельцы (мульти-юзер задел; в MVP всегда одна строка)
-- Все поля кроме telegram_id и created_at могут быть NULL до завершения /start.
CREATE TABLE owners (
    telegram_id        INTEGER PRIMARY KEY,
    jina_api_key       TEXT,
    deepgram_api_key   TEXT,
    openrouter_key     TEXT,
    primary_model      TEXT,
    fallback_model     TEXT,
    github_token       TEXT,
    github_mirror_repo TEXT,                -- "user/soroka-data"
    vps_host           TEXT,                -- для команды /mcp
    vps_user           TEXT,                -- для команды /mcp
    inbox_chat_id      INTEGER,             -- ID канала «Избранное 2»
    setup_step         TEXT,                -- состояние мастера /start: jina|deepgram|openrouter|models|github|channel|done
    created_at         INTEGER NOT NULL
);

-- Записи в базе
CREATE TABLE notes (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    owner_id        INTEGER NOT NULL REFERENCES owners(telegram_id),
    tg_message_id   INTEGER NOT NULL,    -- ID сообщения в канале
    tg_chat_id      INTEGER NOT NULL,
    kind            TEXT NOT NULL,       -- text/voice/youtube/web/pdf/docx/xlsx/image
    title           TEXT,                -- автоизвлечённый или первая строка
    content         TEXT NOT NULL,       -- извлечённый текст для FTS
    source_url      TEXT,                -- если ссылка
    raw_caption     TEXT,                -- caption из ТГ если был
    created_at      INTEGER NOT NULL,
    UNIQUE(owner_id, tg_chat_id, tg_message_id)
);

-- Вложения (файлы скачанные на VPS)
CREATE TABLE attachments (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    note_id         INTEGER NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    file_path       TEXT NOT NULL,       -- относительный путь в data/attachments/
    file_size       INTEGER NOT NULL,
    mime_type       TEXT,
    original_name   TEXT,
    is_oversized    INTEGER DEFAULT 0    -- 1 = только метаданные, файл >20MB
);

-- Полнотекстовый индекс
CREATE VIRTUAL TABLE notes_fts USING fts5(
    title, content, raw_caption,
    content='notes',
    content_rowid='id',
    tokenize='unicode61 remove_diacritics 0'
);

-- Векторный индекс (sqlite-vec)
CREATE VIRTUAL TABLE notes_vec USING vec0(
    note_id INTEGER PRIMARY KEY,
    embedding FLOAT[1024]      -- Jina v3 = 1024 dims
);
```

Файл БД на VPS: `data/soroka.db`, права `chmod 600`. Все ключи в БД хранятся в открытом виде. Аргументация: и `.env`, и БД на одном диске VPS — шифрование БД «театр», даёт ту же защиту что и `chmod 600`. Если позже передумаем — миграция тривиальная.

### 3.3. Pipeline сохранения (`core/ingest.py`)

```
1. handler канала ловит апдейт
2. ставит реакцию 🔄
3. определяет kind (text/voice/url/document/photo)
4. вызывает соответствующий extractor → текст
   - если файл >20MB: пропускаем скачивание, kind=oversized, ставим ⚠️
5. вызывает Jina embeddings → 1024-вектор
6. INSERT в notes + notes_fts (через триггер) + notes_vec
7. ставит реакцию ✅ (или ❌ при исключении)
```

**LLM в этом пайплайне не используется.** Это критично для бюджета и отсутствия галлюцинаций.

### 3.4. Pipeline поиска (`core/search.py`)

```
1. запрос пользователя (текст или транскрипт голоса)
2. core.intent.parse(query) — LLM → {clean_query, filters: {kind?, period?}}
3. BM25: SELECT FROM notes_fts MATCH clean_query LIMIT 30
4. Dense: Jina embed(clean_query) → vec_search в notes_vec LIMIT 30
5. RRF объединяет рейтинги → top-15
6. LLM rerank: «вот запрос, вот 15 фрагментов, верни ID топ-5 в порядке релевантности»
7. для каждого ID: формируем ответ с tg-ссылкой на оригинальное сообщение
   ссылка вида https://t.me/c/<chat_id_short>/<message_id>
```

**Use of LLM:** только шаги 2 и 6, обе операции укладываются в одну `:free` модель.

### 3.5. `/export` — стратегия размера

Telegram bot API лимит на отправку **одного файла** через `sendDocument` — 50MB. Это лимит на каждый файл, а не суммарный.

```
1. собираем zip с содержимым:
   - notes.json (вся таблица notes)
   - schema.sql (CREATE TABLE из БД)
   - soroka.db (полный SQLite)
   - attachments/<note_id>/<filename> (все файлы из data/attachments/)
   - README.md (как разворачивать локально)
2. если zip ≤ 50MB:
   bot.send_document(chat=DM, file=zip)
3. если zip > 50MB И настроен GitHub-зеркало:
   - собираем zip-lite (только soroka.db + notes.json, без вложений) → отправляем в DM
   - полный zip заливаем как GitHub Release в репо-зеркало
   - отправляем в DM ссылку на релиз
4. если zip > 50MB И зеркало НЕ настроено:
   - бот пишет: «Архив 87MB не помещается в Telegram. Настрой GitHub-зеркало через /setgithub.»
   - отправляет zip-lite в DM как минимальный бэкап
```

**GitHub-зеркало** — приватный репозиторий пользователя в его GitHub-аккаунте (например, `andrey/soroka-data`). Настраивается на шаге 5 мастера `/start` или командой `/setgithub`. Это **отдельный репозиторий**, не тот публичный, в котором лежит код Soroka.

**Что попадает в зеркало:**
- При каждом `/export` с zip >50MB → новый GitHub Release с приложенным архивом
- В файлах репозитория: ничего (используем только Releases для бинарников)

### 3.6. MCP-сервер (`src/mcp/server.py`)

Запускается **по требованию через SSH stdio**, не как всегда работающий сервис. На VPS установлен бинарник-обёртка `/usr/local/bin/soroka-mcp`, который делает `docker exec -i soroka-bot python -m src.mcp.server`.

**MCP tools:**

| Tool | Параметры | Возвращает |
|------|-----------|------------|
| `search` | `query: str, limit: int = 5` | top-N с заголовком, фрагментом, ссылкой и ID |
| `get_by_id` | `note_id: int` | полный текст записи + метаданные |
| `list_recent` | `limit: int = 20, kind?: str` | последние N записей |
| `get_attachment` | `note_id: int` | base64-содержимое файла или ошибка `oversized` |

**Команда `/mcp` в боте** печатает:
```
Скопируй этот блок в файл claude_desktop_config.json
(на маке: ~/Library/Application Support/Claude/claude_desktop_config.json)

{
  "mcpServers": {
    "soroka": {
      "command": "ssh",
      "args": ["andrey@123.45.67.89", "soroka-mcp"]
    }
  }
}

Затем перезапусти Claude Desktop.
Готово — теперь Claude видит твою базу через инструмент `soroka`.
```

IP и SSH-юзер берутся из `.env` (`VPS_HOST`, `VPS_USER`).

**Безопасность:** ноль открытых портов в интернет, авторизация — SSH-ключом, который у владельца уже есть для управления VPS.

### 3.7. OpenRouter — выбор моделей

**Источник списка:** `GET https://openrouter.ai/api/v1/models` (без ключа). Берём поля `id`, `name`, `pricing.prompt`, `pricing.completion`, `context_length`.

**Сортировка:** сначала бесплатные (суффикс `:free`), потом по `pricing.prompt` ASC.

**Отображение:** inline-кнопки по 5 моделей на страницу, навигация ◀️ ▶️. Формат кнопки:
```
🆓 google/gemini-2.0-flash-exp:free
   $0.10/1M  meta-llama/llama-3.3-70b-instruct
   $0.80/1M  anthropic/claude-3.5-haiku
```

**Сценарий выбора:**
1. `/models` → бот: «Выбери основную модель» + список с кнопками
2. Клик на модель → бот сохраняет как primary, спрашивает: «Теперь выбери fallback — на случай если основная упадёт» + тот же список
3. Клик → fallback сохранён → бот: «Готово. Модели сохранены.»

**Применение:** модель читается из БД на каждый запрос, перезагрузка контейнера не нужна.

**Fallback-логика в `adapters/openrouter.py`:**
```python
try:
    return call(primary_model, messages)
except (RateLimit, ServerError, Timeout) as e:
    log.warning(f"primary failed: {e}, falling back")
    return call(fallback_model, messages)
```

### 3.8. Онбординг владельца (`/start`)

Срабатывает при первой `/start` от пользователя с `telegram_id == OWNER_TELEGRAM_ID` (из `.env`). После установки на VPS бот **не делает ничего** до тех пор, пока владелец не пройдёт мастер. Шаги — последовательный диалог в DM, состояние хранится в `owners.setup_step`. На каждом шаге сообщение от пользователя обрабатывается как ответ на текущий шаг (а не как поисковый запрос).

**6 шагов:**

1. **Jina API key** (обязательно)
   - Бот: «Привет! Я Soroka. Настроим за 5 минут. Шаг 1/6 — ключ для эмбеддингов. Зарегистрируйся на jina.ai → API → Free tier (1M токенов бесплатно), скопируй ключ и пришли сюда.»
   - Валидация: `POST https://api.jina.ai/v1/embeddings` тестовым запросом
   - Сохраняем в `owners.jina_api_key`

2. **Deepgram API key** (обязательно)
   - «Шаг 2/6 — ключ для распознавания голоса. Зарегистрируйся на deepgram.com (бесплатные $200), создай API key и пришли сюда.»
   - Валидация: `GET https://api.deepgram.com/v1/projects`
   - Сохраняем в `owners.deepgram_api_key`

3. **OpenRouter API key** (обязательно)
   - «Шаг 3/6 — ключ OpenRouter для LLM. Зарегистрируйся на openrouter.ai/keys и пришли сюда.»
   - Валидация: `GET https://openrouter.ai/api/v1/auth/key`
   - Сохраняем в `owners.openrouter_key`

4. **Выбор моделей** (обязательно)
   - Бот тянет список моделей через `GET https://openrouter.ai/api/v1/models`, сортирует (free → дешёвые → дорогие), показывает inline-кнопки по 5 на страницу с пагинацией
   - Пользователь выбирает основную → бот: «Теперь fallback» → выбор fallback
   - Сохраняем в `owners.primary_model`, `owners.fallback_model`

5. **GitHub-зеркало** (рекомендуется, можно пропустить через `/skip`)
   - «Шаг 5/6 — резервное копирование. Создай **приватный** репозиторий вида `<твой-юзер>/soroka-data` на github.com/new. Затем сгенерируй Personal Access Token на github.com/settings/tokens/new с правами только `repo`. Пришли мне токен и имя репо одной строкой: `ghp_xxx user/repo`.»
   - Валидация: `GET https://api.github.com/repos/<repo>` с этим токеном → должен вернуть 200 и `private: true`
   - Сохраняем в `owners.github_token`, `owners.github_mirror_repo`
   - Если `/skip`: предупреждение что `/export >50MB` работать не будет

6. **Канал-инбокс** (обязательно)
   - «Шаг 6/6 — канал. Создай в Telegram **приватный** канал «Избранное 2», добавь меня админом с правами `Post Messages` и `Add Reactions`. Затем форварднь любое сообщение **из этого канала** сюда в DM.»
   - Бот ловит forward (`forward_from_chat`), извлекает `chat_id`, сохраняет в `owners.inbox_chat_id`
   - Сразу публикует тестовое сообщение в канал «✅ Soroka подключилась. Кидай сюда что угодно — индексирую.» и удаляет его через 10 сек

После 6 шага: `setup_step = 'done'`, бот: «Готово! Команды: /help, /status, /export, /mcp. Кидай в канал что угодно — я индексирую. Чтобы найти — пиши прямо сюда.»

**Возобновление мастера:** если пользователь прервал диалог (например, отправил `/cancel` или просто закрыл чат), повторный `/start` возобновит с того шага, на котором остановились.

**Изменение настроек после мастера:** через одиночные команды `/setjina`, `/setdeepgram`, `/setkey`, `/models`, `/setgithub`, `/setvps`, `/setinbox`. Каждая запускает мини-диалог из одного-двух шагов.

## 4. Stack

| Компонент | Выбор | Почему |
|-----------|-------|--------|
| Язык | Python 3.12 | Богатые библиотеки для извлечения, простота кода |
| Telegram | `python-telegram-bot` v22 | Стандарт |
| БД | SQLite + FTS5 + `sqlite-vec` | Один файл, всё в одном месте, бэкап = `cp` |
| Embeddings | Jina v3 (1024d) | Multilingual, EU, не блокирует RU |
| LLM (rerank+intent) | OpenRouter, выбор пользователем | Free-модели достаточны |
| Голос → текст | Deepgram | $200 free tier хватит надолго |
| Веб-извлечение | `trafilatura` | Чистый текст без шума |
| YouTube | `yt-dlp` | Транскрипт через `--write-auto-sub` |
| OCR | `tesseract` (`rus+eng`) | Локально, без API |
| Хостинг | EU VPS (любой ≥1GB RAM) | OpenRouter блокирует RU IP |
| Контейнеризация | Docker + `docker-compose` | Один бот-сервис, файл БД через volume |

## 5. Безопасность и секреты

### 5.1. Где живут какие секреты

**Главный принцип:** в `.env` — только **минимум для запуска бота**. Всё остальное собирается через Telegram-мастер `/start` и сохраняется в SQLite. Это позволяет владельцу настроить всё через чат, не редактируя файлы на сервере.

| Секрет | Где | Когда задаётся |
|--------|-----|----------------|
| `TELEGRAM_BOT_TOKEN` | `.env` | Один раз при установке (его требует сам процесс бота при старте) |
| `OWNER_TELEGRAM_ID` | `.env` | Один раз при установке (для проверки «свой/чужой») |
| `JINA_API_KEY` | SQLite | Шаг 1 мастера `/start` |
| `DEEPGRAM_API_KEY` | SQLite | Шаг 2 мастера `/start` |
| `OPENROUTER_KEY` | SQLite | Шаг 3 мастера `/start` |
| `primary_model`, `fallback_model` | SQLite | Шаг 4 мастера `/start` |
| `GITHUB_TOKEN` + `GITHUB_MIRROR_REPO` | SQLite | Шаг 5 мастера `/start` |
| `inbox_chat_id` | SQLite | Шаг 6 мастера `/start` |
| `vps_host`, `vps_user` | SQLite | Команда `/setvps` (для генерации MCP-конфига) |

**Что это даёт:**
- При установке пользователь редактирует `.env` ровно один раз (через `./bin/install`, который пишет файл сам)
- Все смены ключей и моделей — через бот без SSH
- На раздачу аудитории — простая инструкция: «запусти `./bin/install`, потом отправь боту `/start`»

### 5.2. `.gitignore` (что не летит в репозиторий)

```
.env
data/
*.db
*.db-journal
*.db-shm
*.db-wal
attachments/
__pycache__/
*.pyc
.venv/
.pytest_cache/
```

В репозитории только: код + `.env.example` с пустыми заглушками + `docker-compose.yml` + `Dockerfile` + документация.

### 5.3. Два разных GitHub-репозитория

**Репозиторий с кодом** (`github.com/<user>/soroka`):
- Может быть публичным — код пригоден для раздачи аудитории
- Никогда не содержит `.env`, БД, вложений

**Репозиторий-зеркало** (опциональный, для `/export`):
- Обязательно приватный
- Содержит выгрузки БД и вложений
- В `.env` на VPS: `GITHUB_MIRROR_REPO=user/soroka-data` + `GITHUB_TOKEN` с правами только на этот один репо

### 5.4. SSH

Доступ к VPS по SSH — обязателен. Используется для:
- Первоначального развёртывания (`git clone`, `docker compose up`)
- Обновлений (`git pull && docker compose up -d --build`)
- MCP-сервера (`ssh user@vps soroka-mcp` — Claude Desktop делает это автоматически)

Авторизация — только по SSH-ключу (парольная авторизация выключается на VPS как часть базовой настройки). Никаких других открытых портов в интернет, кроме 22.

### 5.5. Минимальные требования VPS

- 1GB RAM, 20GB SSD, EU локация
- Ubuntu 22.04 + Docker
- Открытый порт 22 (SSH) — больше ничего

## 6. Развёртывание

### 6.1. `.env` (минимум)

Файл создаётся автоматически скриптом `./bin/install`. Пользователь руками его не редактирует.

```
TELEGRAM_BOT_TOKEN=1234567890:AAH...
OWNER_TELEGRAM_ID=123456789
```

Всё остальное собирается мастером `/start` в Telegram и хранится в SQLite (см. §5.1).

### 6.2. `docker-compose.yml`

```yaml
services:
  bot:
    build: .
    container_name: soroka-bot
    restart: unless-stopped
    env_file: .env
    volumes:
      - ./data:/app/data
    command: python -m src.bot.main
```

**MCP-сервер не отдельный контейнер** — это команда внутри того же образа, вызываемая через `docker exec` из хостового скрипта.

### 6.3. Хостовый скрипт `/usr/local/bin/soroka-mcp`

```sh
#!/bin/sh
exec docker exec -i soroka-bot python -m src.mcp.server
```

Кладётся в `/usr/local/bin/` скриптом установки.

### 6.4. Скрипт `./bin/install` (главный путь установки)

Запускается **на ноутбуке пользователя** (Mac/Linux). Поднимает бота на VPS целиком, не заставляя редактировать файлы на сервере.

**Интерактивный сценарий:**

```
$ git clone https://github.com/<user>/soroka.git
$ cd soroka
$ ./bin/install

🐦 Soroka — установщик

▸ IP-адрес VPS: 65.21.45.122
▸ SSH-пользователь [root]: <enter>
▸ Путь к SSH-ключу [~/.ssh/id_rsa]: <enter>

Проверяю подключение... ✓

▸ Telegram-токен (получи у @BotFather):
  1234567890:AAHr...
▸ Твой Telegram ID (узнай у @userinfobot):
  987654321

Развёртываю на VPS:
  [✓] Установка Docker и Docker Compose
  [✓] Клонирование репозитория в /opt/soroka
  [✓] Запись /opt/soroka/.env (chmod 600)
  [✓] docker compose up -d
  [✓] Установка /usr/local/bin/soroka-mcp

Готово! Открой Telegram, найди своего бота и отправь /start
```

**Под капотом** — Bash-скрипт ~80 строк, который через SSH выполняет heredoc:

```bash
ssh "$USER@$IP" bash <<EOF
  apt-get update && apt-get install -y docker.io docker-compose-plugin git
  mkdir -p /opt && cd /opt
  [ -d soroka ] || git clone https://github.com/<user>/soroka.git
  cd soroka
  cat > .env <<ENV
TELEGRAM_BOT_TOKEN=$TG_TOKEN
OWNER_TELEGRAM_ID=$OWNER_ID
ENV
  chmod 600 .env
  docker compose up -d --build
  cp scripts/soroka-mcp /usr/local/bin/
  chmod +x /usr/local/bin/soroka-mcp
EOF
```

**Неинтерактивный режим (для агентов):**
```
./bin/install \
  --vps-ip=65.21.45.122 \
  --ssh-user=root \
  --ssh-key=~/.ssh/id_rsa \
  --tg-token='1234:ABC...' \
  --owner-id=987654321
```

### 6.5. `AGENTS.md` — инструкция для AI-агентов

В корне репозитория. Структурированный README специально для Claude Code, Cursor и подобных. Содержит:

- Список переменных, которые агент должен запросить у пользователя
- Точные команды развёртывания (флаги `./bin/install`)
- Команды для проверки состояния (`docker ps`, `docker logs soroka-bot`, путь к БД)
- Как читать состояние мастера: `sqlite3 /opt/soroka/data/soroka.db "SELECT setup_step FROM owners"`
- Что делать после установки: «открой Telegram → отправь `/start`, остальное — диалог в боте»
- Список команд бота для диагностики (`/status`, `/help`)

Агент получает чёткий протокол: «спроси у юзера N значений, выполни 1 команду, скажи юзеру открыть Telegram». Никаких вариативных шагов на сервере.

### 6.6. Обновление

```
$ ./bin/update <vps-ip>
```

Скрипт делает SSH в VPS, выполняет `cd /opt/soroka && git pull && docker compose up -d --build`. БД и `.env` не трогаются.

## 7. Что НЕ входит в MVP

Фиксирую сразу, чтобы не было соблазна:

- ❌ Тэги, категории, топики, таксономия
- ❌ Веб-интерфейс
- ❌ Несколько владельцев в одном инстансе (схема `owners` готова на мульти-владельца, но проверка `telegram_id == OWNER_TELEGRAM_ID` — каждый ставит свой инстанс)
- ❌ Локальные эмбеддинги (только Jina API)
- ❌ Local Bot API Server для файлов >20MB (можно добавить позже отдельным сервисом)
- ❌ Автокатегоризация LLM, автотэги
- ❌ Граф связей между записями
- ❌ Удаление/редактирование записей через бота (можно сделать командой `/delete <id>` если попросят)
- ❌ Автоматический бэкап на GitHub по расписанию (только при `/export >50MB`)
- ❌ Шифрование БД (полагаемся на права `chmod 600` и безопасность VPS)
- ❌ Web-installer / .deb-пакет (только Bash-скрипт)
